目录层

2026年02月14日 1.4k 字 大概 6 分钟

你想理解 xv6 目录层的核心实现逻辑——目录如何以“特殊文件”的形式存储，以及 dirlookup/dirlink 这两个核心函数如何实现目录的查找和条目添加，我会拆解目录的存储结构、函数逻辑和关键设计考量（比如死锁规避）。

一、目录层的核心设计：目录是“特殊文件”

xv6 中目录本质是类型为 T_DIR 的 inode，和普通文件的核心区别仅在于：

  1. inode 的 type 字段为 T_DIR（普通文件是 T_FILE）；
  2. 数据块中存储的不是普通数据，而是目录条目（struct dirent） 数组。

1. 目录条目结构（struct dirent）

  • 关键规则：inum=0 表示该条目空闲，可用于新增；名称长度不足 14 字节时，末尾补 \0。
  • 存储方式：多个 dirent 连续存储在目录 inode 的数据块中，和普通文件的字节流存储逻辑一致。

2. 目录的默认条目

每个目录创建时会默认包含两个条目，保证路径查找的合法性：

  • .：指向目录自身的 inode；
  • ..：指向父目录的 inode。

二、核心函数解析（dirlookup + dirlink）

1. dirlookup：在目录中查找指定名称的条目

作用：在目录 dp 中搜索名称为 name 的条目，找到则返回对应 inode 指针，同时返回条目在目录中的字节偏移（供编辑用）。

  • 核心设计：返回未锁定的 inode（死锁规避）
    这是 dirlookup 最关键的设计，原因：
    1. 调用 dirlookup 时，调用者已经锁定了目录 inode dp；
    2. 如果查找的是 .（当前目录），若 dirlookup 返回锁定的 inode，会导致调用者重复锁定 dp，触发死锁；
    3. 若查找 ..（父目录），多进程场景下也可能因嵌套锁定导致死锁；
    4. 解决方案：返回未锁定的 inode，调用者需先解锁 dp，再锁定返回的 inode，保证一次只持有一个锁。

2. dirlink：向目录添加新的条目

作用：将 name 和 inum 对应的条目写入目录 dp，若名称已存在则返回错误。

  • 核心逻辑：
    1. 先检查重名：调用 dirlookup，若找到同名条目直接返回错误；
    2. 找空闲位置：优先复用 inum=0 的空闲条目，无空闲则在目录末尾（dp->size 处）新增；
    3. 写入条目：通过 writei 写入，若写入位置超出目录当前大小，writei 会自动扩展目录 inode 的 size；
    4. 事务约束：需包裹在 begin_op()/end_op() 中，因为修改了目录 inode 的数据块。

四、关键细节总结

  1. 目录的本质：是 type=T_DIR 的 inode，数据块存储 dirent 条目，复用普通文件的 readi/writei 读写；
  2. dirlookup 核心：遍历条目匹配名称，返回未锁定的 inode 避免死锁；
  3. dirlink 核心：先查重名，再复用空闲条目（无则新增），通过 writei 写入；
  4. 空条目规则：inum=0 标识空闲，优先复用减少目录碎片；
  5. 死锁规避：dirlookup 返回未锁定 inode，调用者需“解锁旧锁→锁定新锁”，一次只持一个锁。

总结

xv6 目录层的核心关键点：

  1. 特殊文件设计：目录复用 inode 层的所有机制（块管理、读写、锁），仅通过 type=T_DIR 和 dirent 条目区分；
  2. 核心函数分工：dirlookup 查条目（避死锁），dirlink 加条目（查重名）；
  3. 空闲复用：优先复用 inum=0 的空条目，减少目录碎片；
  4. 事务约束：所有目录操作需包裹在日志事务中，保证崩溃一致性。

一句话记忆：目录是T_DIR类型的文件，dirlookup查条目返未锁inode，dirlink加条目先查重名再复用空闲。
