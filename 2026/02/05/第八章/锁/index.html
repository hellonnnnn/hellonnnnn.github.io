锁

2026年02月05日 1.6k 字 大概 7 分钟

一、为什么“看起来正确”的 acquire 是错的？

朴素实现：

❌ 问题本质

在 多核 情况下，这两步不是原子的：

  1. 读 lk->locked
  2. 写 lk->locked

可能发生：

  • CPU0 读到 locked == 0
  • CPU1 读到 locked == 0
  • 两个 CPU 都写 locked = 1

👉 两个 CPU 同时“拿到”锁
👉 互斥性（mutual exclusion）被破坏

📌 结论：

“检查 + 设置”必须是一个不可分割的原子操作

二、硬件是如何解决这个问题的？

RISC-V 提供的原子指令：amoswap

语义（一次完成）：

关键点：

  • 原子性保证

  • 在 swap 期间：

    • 其他 CPU 无法插入访问

  • 等价于：

    • “测试并设置（test-and-set）”

三、xv6 的 acquire 是如何实现的？

xv6 并不直接写汇编，而是用：

这个函数的语义

  • 原子地：

    • 把 lk->locked 设为 1
    • 返回 旧值

等价于 RISC-V 的 amoswap

acquire 的核心逻辑（抽象版）

行为分析：

旧值 含义 结果
0 锁空闲 当前 CPU 成功获取锁
1 锁被占用 继续自旋

📌 即使 swap 失败：

  • 仍然是原子的
  • 不会破坏锁状态

为什么叫“自旋锁”（spinlock）？

  • 没有睡眠
  • 不让出 CPU
  • 一直在 while 循环里“空转等待”

👉 适合：

  • 临界区短
  • 内核态
  • 不能睡眠的场景

四、为什么 release 也不能用普通赋值？

你可能会觉得：

就够了。

❌ 但 C 语言标准不保证：

  • 赋值是原子的
  • 编译器不会拆成多个 store
  • 不会乱序重排

在并发代码中，这是 未定义风险。

xv6 的 release 实现

作用：

  • 原子地 将 locked 设为 0
  • 同样可归结为 amoswap

📌 顺序很重要：

  1. 清除 lk->cpu
  2. 再释放锁

保证：

  • 其他 CPU 拿到锁时
  • 锁状态是完全一致的

五、lk->cpu 字段是干什么的？

用途：

  • 调试

  • 检测：

    • 重复加锁
    • 非持有者释放锁

重要原则：

lk->cpu 也是共享数据，只能在持有锁时访问

这是一个非常经典的“锁保护锁自身元数据”的例子。

六、acquire / release 的本质总结

acquire 做了三件事：

  1. 原子 test-and-set
  2. 自旋等待
  3. 记录持有锁的 CPU（调试）

release 做了两件事：

  1. 清理调试信息
  2. 原子释放锁

七、面试必考点（直接背）

Q1：为什么 acquire 不能用 if + 赋值？

因为在多核环境下，读和写不是原子的，可能导致多个 CPU 同时获得锁，破坏互斥性。

Q2：xv6 的自旋锁依赖什么硬件机制？

依赖 CPU 提供的原子指令，在 RISC-V 上是 amoswap，用于实现 test-and-set。

Q3：为什么 release 也要用原子操作？

因为 C 语言普通赋值在并发场景下不保证原子性，编译器可能拆分或重排指令。

Q4：什么场景适合自旋锁？

  • 临界区短
  • 不允许睡眠
  • 内核态
  • CPU 数量有限

Q5：自旋锁的缺点？

  • 浪费 CPU
  • 临界区长时性能极差
  • 不适合用户态或 I/O 等待

xv6 睡眠锁（sleep-locks）整理

一、为什么自旋锁不够用？

自旋锁像一个原地打转的陀螺：
没拿到锁，就一直转 CPU。

自旋锁的两个致命问题

1️⃣ 临界区太长会浪费 CPU

  • 文件系统读写磁盘

  • 磁盘 I/O 可能 几十毫秒

  • 如果用自旋锁：

    • 其他 CPU 一直自旋
    • CPU 时间被白白烧掉

2️⃣ 持有自旋锁时不能让出 CPU

原因有三层：

  • acquire 是自旋，不会主动让出 CPU

  • 如果持锁进程 yield：

    • 别的进程自旋
    • 持锁进程反而得不到 CPU
    • ⇒ 死锁

  • xv6 规定：
    持有自旋锁时必须关中断

    • yield / sleep 都需要中断
    • 逻辑上不允许

📌 结论：

自旋锁只适合 短、快、不可睡眠 的临界区

二、睡眠锁的设计目标

我们想要一种锁：

  • 等锁时 不浪费 CPU

  • 拿不到锁时 sleep

  • 持锁时：

    • 可以 sleep
    • 可以开中断

  • 适合 慢操作（磁盘、文件系统）

于是有了：sleep-lock

sleep() 会 原子地：

  • 释放自旋锁

这一步避免了经典的：

3️⃣ 效果是什么？

  • 等锁的进程不自旋
  • CPU 可运行其他进程
  • 锁真正变成“阻塞式”的

四、睡眠锁的使用限制（非常重要）

❌ 1. 不能用于中断处理函数

原因：

  • acquiresleep 可能 sleep
  • 中断上下文 不能 sleep
  • 中断必须尽快返回

📌 所以：

中断里只能用自旋锁

❌ 2. 不能在自旋锁临界区中使用

错误示例（概念）：

原因：

  • 自旋锁持有期间：

    • 中断关闭
    • 不允许 sleep

  • acquiresleep 可能让出 CPU

  • 直接破坏自旋锁语义

✅ 3. 反过来是允许的

解释：

  • 睡眠锁期间中断是开着的
  • 临时进入一个短的自旋锁临界区是安全的

五、自旋锁 vs 睡眠锁（对照表）

维度 自旋锁 睡眠锁
等待方式 忙等（自旋） sleep
是否浪费 CPU 是 否
是否允许 sleep ❌ ✅
是否关中断 是 否
能否用于中断 ✅ ❌
适合场景 短临界区 长操作（I/O）

六、什么时候用哪种锁？（经验法则）

用自旋锁：

  • 临界区很短
  • 只做内存操作
  • 可能在中断上下文
  • 必须快速完成

例子：

  • ticks
  • 进程状态
  • 就绪队列
  • 缓冲区元数据

用睡眠锁：

  • 可能等待 I/O
  • 可能 sleep
  • 操作耗时不可控

例子：

  • inode
  • 文件内容
  • 磁盘读写路径

七、面试总结版（背这一段就够）

xv6 使用自旋锁保护短临界区，避免上下文切换开销；
对于可能阻塞或耗时较长的操作，引入睡眠锁，使等待进程 sleep 让出 CPU；
睡眠锁内部仍由自旋锁保护状态，但 acquiresleep 会原子地释放自旋锁并休眠；
因此睡眠锁不能用于中断处理，也不能嵌套在自旋锁临界区中。
