锁和中断处理函数

2026年02月05日 1k 字 大概 4 分钟

这一节其实在回答一个非常尖锐的问题：

为什么 xv6 在拿自旋锁时一定要关中断？而且顺序还不能错？

我按 问题 → 根因 → xv6 的设计 → 顺序为什么不能颠倒 → 面试总结 来讲，帮你把逻辑彻底“拧直”。

一、问题的根源：锁 + 中断 = 同一 CPU 的“自我死锁”

关键前提（非常重要）

  • 中断处理函数也是在当前 CPU 上执行的

  • 中断不是“另一个线程”，而是：

    当前代码被强行打断，在同一个 CPU上切换执行

经典死锁场景（你文中那个例子）

场景

  • ticks 是共享变量

  • 被：

    • 普通内核线程（sys_sleep）
    • 定时器中断处理函数（clockintr）
      同时访问

  • 用 tickslock 保护

死锁时序（一步一步）

  1. CPU0 正在执行 sys_sleep

  2. sys_sleep：

  3. 此时 CPU0 持有 tickslock

  4. 突然，定时器中断到来

  5. CPU0 进入 clockintr

  6. clockintr 也尝试：

  7. 发现锁被占用 → 自旋等待

💥 问题来了：

  • tickslock 的释放者是 sys_sleep

  • 但 sys_sleep：

    • 被中断打断
    • 只有等 clockintr 返回才能继续

👉 同一 CPU 上形成了“自己等自己”的死锁

二、核心结论（一句话版）

如果一个自旋锁可能在中断处理函数中被使用，那么在持有该锁时，绝不能允许本 CPU 接收中断。

否则：

  • 中断处理函数可能尝试获取同一把锁
  • 直接死锁

三、xv6 的解决方案（非常保守，但极安全）

xv6 的策略不是“只对部分锁关中断”，而是：

只要 acquire 任意自旋锁，就关中断

这比“必要条件”更严格，但好处是：

  • 简化推理
  • 不容易写错
  • 教学系统里非常合理

四、push_off / pop_off 在干什么？

1️⃣ 为什么不能直接 intr_off / intr_on？

因为可能存在 嵌套临界区：

如果：

  • 第一个 acquire 关中断
  • 第二个 acquire 又关一次
  • 第一个 release 就开中断

👉 中断状态会乱套

2️⃣ push_off / pop_off 的作用

它们做了三件事：

  1. 记录原来的中断状态
  2. 维护一个“关中断嵌套计数”
  3. 只在最外层 release 时恢复中断状态

六、为什么“别的 CPU”不会有这个问题？

你文中也点到了一个非常重要的点：

中断仍然可能发生在其他 CPU 上

这是安全的，因为：

  • CPU1 的中断处理函数自旋等待
  • CPU0 可以继续运行并释放锁
  • 不存在“自己等自己”

👉 死锁只发生在“同一 CPU”

七、面试级总结（可以直接背）

Q1：为什么自旋锁和中断会产生死锁？

因为中断处理函数可能在同一 CPU 上尝试获取线程已经持有的自旋锁，导致线程无法继续执行释放锁，从而自我死锁。

Q2：xv6 是如何避免这个问题的？

xv6 在获取任何自旋锁时都会禁用当前 CPU 的中断，并在释放最外层锁时恢复中断。

Q3：为什么 acquire 必须先关中断再设置锁？

否则会出现一个“持有锁但中断仍开启”的窗口，中断处理函数可能在该窗口内触发并尝试获取同一把锁，导致死锁。
