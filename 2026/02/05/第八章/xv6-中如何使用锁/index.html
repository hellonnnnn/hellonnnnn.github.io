xv6 中如何使用锁

2026年02月05日 962 字 大概 4 分钟

一、什么时候必须用锁？（必要条件）

⭐ 第一条铁律（最重要）

任何变量，只要满足：

  • 可能被一个 CPU 写
  • 同时可能被另一个 CPU 读或写
    👉 就必须加锁

这是判断要不要锁的最低标准。

⭐ 第二条铁律：锁保护的是“不变量”

锁不是为了“防止同时执行代码”，
而是为了 保护共享数据的不变量（invariants）

如果一个不变量涉及：

  • 多个字段
  • 多个内存位置

👉 它们通常必须由同一把锁保护

否则：

  • 一个 CPU 正在“中间态”
  • 另一个 CPU 读到“半成品”
  • 立刻出 bug

二、什么时候“不该”用锁？（性能视角）

锁虽然保证正确性，但一定降低并行性。

极端方案：大内核锁（Big Kernel Lock）

思路：

优点：

  • 设计简单
  • 易于从单核内核迁移

缺点：

  • ❌ 一次只能一个 CPU 在内核
  • ❌ 多核 ≈ 单核性能

📌 xv6 没有采用大内核锁，而是使用多把细粒度锁

三、锁粒度的取舍：粗 vs 细

这是内核设计的核心难题之一。

1️⃣ 粗粒度锁（coarse-grained locking）

xv6 示例：kmem.lock

  • 一个全局空闲页链表
  • 一个锁保护整个分配器

问题：

  • 自旋是纯浪费 CPU
  • 高并发时性能急剧下降

📌 优化方向（理论上）：

  • 每 CPU 一个 freelist
  • 每个 freelist 一把锁
  • 减少锁争用

2️⃣ 细粒度锁（fine-grained locking）

xv6 示例：文件 / inode / buffer

  • 每个文件一个锁
  • 每个 inode 一个锁
  • 每个 buffer 一个锁

好处：

  • 操作不同对象的进程可以真正并行
  • 极大提升吞吐量

代价：

  • 设计复杂
  • 易出死锁
  • 需要明确锁顺序

📌 最终取舍必须靠性能测试 + 复杂度评估

四、xv6 中所有关键锁（表 6.3 的“含义版”）

下面不是死记表格，而是理解每把锁在保护什么不变量。

🔹 内存与资源分配

  • kmem.lock
    👉 保护物理页空闲链表

  • pid_lock
    👉 保护 next_pid 单调递增不变量

  • ftable.lock
    👉 保护文件结构体的分配与回收

🔹 缓存与文件系统

  • bcache.lock
    👉 保护 block buffer cache 的分配

  • b->lock
    👉 保护单个磁盘块缓冲区的状态和内容

  • icache.lock
    👉 保护 inode cache 的分配

  • ip->lock
    👉 保护 inode 的元数据和文件内容

🔹 进程与同步

  • p->lock
    👉 保护进程状态（RUNNING / SLEEPING / ZOMBIE）

  • 管道的 pi->lock
    👉 保护管道缓冲区和读写指针

🔹 I/O 与设备

  • vdisk_lock
    👉 串行化磁盘硬件 + DMA 队列访问

  • cons.lock
    👉 防止控制台输出字符交错

🔹 时间与日志

  • tickslock
    👉 保护全局时钟计数

  • log.lock
    👉 串行化文件系统事务日志

五、面试“必杀级”总结（可直接背）

Q1：如何判断一个变量需不需要锁？

只要它可能被多个 CPU 并发访问，且至少有写操作，就必须使用锁。

Q2：锁真正保护的是什么？

锁保护的是共享数据的不变量，而不是某段代码。

Q3：为什么不能所有地方都加锁？

锁会降低并行性，过度加锁会让多核系统退化成单核性能。

Q4：什么是粗粒度锁和细粒度锁？

  • 粗粒度：简单，但并发性差
  • 细粒度：并发性好，但实现复杂、易出错

Q5：xv6 为什么性能不高？

xv6 为了教学清晰性，采用了较多粗粒度锁，没有做复杂的并行优化。

六、现在这个阶段，真正要掌握的能力

不是背锁表，而是能做到：

✅ 看到一个共享数据结构
→ 说出它的 不变量
→ 判断 哪些字段必须同一把锁保护
→ 知道 锁放大 or 拆分的后果
