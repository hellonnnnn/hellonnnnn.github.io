分布式算法

2026年03月12日 1.9k 字 大概 8 分钟

📝 分布式共识算法：从 Paxos 到 Multi-Paxos

1. Basic Paxos：分布式一致性的基石

Basic Paxos 的目标是让分布式系统中的多个节点对单个值（Single Value）达成一致。

  • 核心流程（两阶段）：
  1. Prepare 阶段：提议者（Proposer）带着全局唯一且递增的编号 $N$ 去询问决策者（Acceptors）。决策者承诺不再接受比 $N$ 小的提议。
  2. Accept 阶段：如果超过半数决策者响应，提议者发送正式值 $V$。如果此时决策者已经投过票，提议者必须“顺从”旧值，否则可以使用自己的新值。

2. Multi-Paxos：为了性能的演进

Basic Paxos 每次只能确定一个值，且由于两阶段交互，网络开销巨大。Multi-Paxos 对此进行了优化，用于处理一系列连续的值（日志流）。

  • 核心改进：引入 Leader：
  1. 通过一次完整的 Paxos 过程选出一个 Leader（主节点）。
  2. 跳过第一阶段：一旦 Leader 确定，后续的多个提案可以直接进入 Accept 阶段，省去了 Prepare 的往返时间（RTT）。
  3. 只有当 Leader 宕机时，才会重新触发第一阶段选主。

❌ 它们的缺点（Limitations）

虽然 Paxos 在理论上是完美的，但在实际工程实现中存在明显的短板：

一、 Basic Paxos 的缺点

  1. 活锁问题（Livelock / Duueling Proposers）：
  • 如果有两个提议者交替发出越来越大的编号 $N$，它们会互相“覆盖”对方在 Prepare 阶段的承诺，导致大家都在争抢，却没有任何一个提案能进入 Accept 阶段。
  1. 效率低下：
  • 达成一个共识需要两次网络往返（Prepare + Accept），在高并发场景下延迟太高。
  1. 难于理解与实现：
  • Paxos 以“晦涩难懂”著称，工程化实现（如处理各种边缘故障）极其复杂。

二、 Multi-Paxos 的缺点

  1. ** Leader 压力过大（瓶颈）**：
  • 所有的读写请求都经过 Leader，Leader 的网络带宽和处理能力会成为整个集群的上限。
  1. 配置变更困难：
  • 在集群运行过程中动态地增加或减少节点（Membership Change），Paxos 论文中并未详细说明，实现起来非常容易出错。
  1. “空洞”问题（Log Gaps）：
  • 如果 Leader 发送了多条指令，其中某一条因为网络丢包没达成共识，而后面的指令达成了共识，会导致日志中出现“空洞”，状态机无法连续执行，增加了恢复的复杂度。

💡 总结对照

特性 Basic Paxos Multi-Paxos
解决问题 单个值的共识 一系列连续操作（日志）的共识
性能 低（多次 RTT） 高（Leader 模式下 1 次 RTT）
稳定性 易产生活锁 较稳定，但依赖 Leader 存活
应用现状 理论基础，极少直接用于工程 是 Raft、ZAB 等算法的灵魂原型

太棒了！既然你对 ZAB 的兴趣已经从“宏观概念”深入到了“具体实现”，那我们就脱下理论的外衣，直接看看 ZooKeeper 在底层代码和工程架构上到底是怎么把 ZAB 落地实现的。

在 ZooKeeper 的实际源码中，ZAB 的实现主要围绕 ZXID 的位运算、Fast Leader Election (FLE) 算法、三种同步策略 以及 底层通信队列 这四个核心模块展开。

1. ZXID 的硬核实现（位运算）

我们在前面提到 ZXID 是一个 64 位的整数。在底层的 Java 实现中，它是通过位运算（Bitwise Operations）来极速处理的，以此保证极高的性能。

  • 结构：高 32 位是 epoch（纪元），低 32 位是 counter（单调递增计数器）。
  • 生成逻辑：
    当新 Leader 产生时，它会从本地日志中取出最大的 ZXID，解析出前任的 epoch。
    新 epoch = 旧 epoch + 1。
    新 ZXID = (新 epoch << 32) | 0 （即将高 32 位赋新值，低 32 位清零）。
  • 比较逻辑：因为是组合成了一个 64 位的 long 型变量，比较两个节点谁的数据更新时，不需要拆开比，直接用 > 或 < 比较这两个 long 型整数即可，这在 CPU 层面执行极快。

2. 具体实现：Fast Leader Election (FLE) 算法

当集群启动或 Leader 宕机时，ZooKeeper 默认使用 FLE 算法在几百毫秒内决出新 Leader。

选票的数据结构：
每个节点在广播选票时，发送的数据包核心包含三个字段：(my_epoch, my_zxid, my_server_id)。

  • my_epoch：我所处的纪元。
  • my_zxid：我本地最后一条提交的事务 ID。
  • my_server_id：我的机器编号（配置文件里的 myid）。

具体的 PK（打擂台）规则：
当节点 A 收到节点 B 的选票时，按以下严格顺序进行比较：

  1. 先比 Epoch：谁的 Epoch 大，谁赢。
  2. 若 Epoch 相同，比 ZXID：谁的 ZXID 大（数据越新），谁赢。
  3. 若 ZXID 还相同，比 Server ID：谁的 myid 大，谁赢（这是一个兜底策略，防止死锁）。

改票与广播：
如果节点 A 发现 B 的选票赢了自己，A 就会修改自己的选票，把自己的票改成 B 的模样，然后再次向全网广播：“我改主意了，我推选 B！”
当某个节点收到了过半数的相同选票，它就知道大局已定，选举结束。

3. 具体实现：数据同步的“三板斧” (Sync Phase)

新 Leader 选出来后，必须把 Followers 参差不齐的数据“拉平”。ZooKeeper 底层实现了三种具体的同步指令：

  • DIFF（差异同步）：

  • 场景：Follower 落后得不多。

  • 实现：Leader 内存里维护了一个提议缓存队列（Proposal Cache）。如果 Follower 缺少的那些提案刚好还在这个缓存里，Leader 就直接把这些缺失的 Proposal 按顺序打包发给 Follower 补齐。

  • TRUNC（截断同步）：

  • 场景：Follower 包含了 Leader 没有的“废弃数据”（比如前任 Leader 刚提议还没 Commit 就挂了，这个 Follower 恰好记录了）。

  • 实现：Leader 会发指令告诉 Follower：“把 ZXID 大于 x 的部分全部切掉（Truncate）！”Follower 会无条件删除本地多余的日志。

  • SNAP（快照同步）：

  • 场景：Follower 落后太多（缓存里找不到历史记录了），或者是一个全新加入集群的节点。

  • 实现：Leader 直接把本地整个内存数据序列化成一个全量快照（Snapshot）文件，通过网络一股脑儿全发给 Follower。Follower 清空自己，直接加载快照。

4. 具体实现：广播阶段的 FIFO 队列

进入正常工作状态后，ZAB 的 2PC（两阶段提交）要保证绝对的顺序，具体在工程上是如何做到不乱序的呢？

  • 单一发送者：所有的写请求不管发给哪个 Follower，底层都会被转发（Forward）给 Leader 处理。
  • TCP 层的顺序保证：ZooKeeper 节点之间使用 TCP 长连接通信，TCP 协议本身保证了数据包不乱序、不丢失。
  • 发件箱机制（Send Queue）：Leader 会为每一个 Follower 分配一个单独的 FIFO 队列。Leader 产生的 Proposal 会按 ZXID 的顺序依次放入这些队列中，然后通过单独的发送线程串行发给 Follower。
  • 不中止的 2PC：与传统的数据库的两阶段提交不同，ZAB 的 2PC 没有回滚（Abort）逻辑。只要过半节点 ACK 了，Leader 就会强行发 Commit。个别慢的节点就算没回复，之后也会通过上面的“数据同步”机制追平。

总结一下：
ZAB 的具体实现是一套环环相扣的工程杰作：利用位运算加速比较，利用 FLE 算法快速破局，利用 DIFF/TRUNC/SNAP 兜底数据一致性，最后利用 TCP+FIFO 队列榨干网络性能。
