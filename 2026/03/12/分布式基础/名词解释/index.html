名词解释

2026年03月12日 1.6k 字 大概 6 分钟

这份技术清单含金量极高！它们构成了现代互联网高并发、分布式、微服务架构的“核心基础设施”。

1. 数据存储与缓存篇 (Storage & Cache)

  • NoSQL (Not Only SQL): 泛指非关系型数据库。打破了传统表格（如 MySQL）的束缚，专为海量数据、高并发读写和灵活的数据结构（键值、文档、图）而生，是 BASE 理论的忠实拥趸。
  • Redis: 地表最强的内存数据结构存储系统。常被用作极速缓存、分布式锁，单线程架构却能扛住超高并发，支持丰富的数据类型（String, List, Set, Hash 等）。
  • Redis Cluster: Redis 的官方分布式去中心化集群方案。抛弃了一致性哈希，采用 16384 个“哈希槽”来实现数据分片，兼顾了高可用与运维的精准掌控。
  • Memcached: 纯粹且极致的内存键值缓存。相比 Redis 的多才多艺，它更简单，只支持 Key-Value，但底层是多线程的，是抵挡关系型数据库读请求压力的纯粹“防弹衣”。

2. 消息传递与分布式协调篇 (Messaging & Coordination)

  • Kafka: 吞吐量极高的分布式消息队列/事件流平台。大数据的“中央神经系统”，专治海量日志收集、高并发下的流量削峰和系统解耦。
  • Zookeeper: 分布式系统的严苛大管家（CP 架构代表）。专门负责分布式协调、配置管理和分布式锁，基于多数派选举（ZAB 协议），以提供顺序一致性著称。
  • Eureka: 微服务架构中的服务注册与发现中心（AP 架构代表）。相当于微服务之间的“通讯录”，秉持高可用原则，网络故障时宁可返回旧的地址列表，也绝不罢工拒绝服务。

3. 网络流量与网关篇 (Networking & Proxy)

  • Nginx: 极高性能的 HTTP 和反向代理服务器。互联网大厂应对用户流量的第一道大门，七层（应用层）负载均衡的王者，处理静态文件和高并发连接的能力冠绝天下。
  • HAProxy: 极其稳定专业的 TCP/HTTP 负载均衡器。不仅能做七层，更常用于四层（传输层）代理，比如放在多台后端数据库（如 MySQL 集群）前面做极其高效的流量分发。
  • BitTorrent (BT): 经典的 P2P (点对点) 文件共享协议。去中心化的典范，下载同一个文件的人越多，您的下载速度反而越快，底层深度依赖分布式哈希表（DHT）。

4. 架构基石与通信基石篇 (Infrastructure & Communication)

  • Docker: 划时代的轻量级容器化技术。把代码和所有依赖的运行环境打包成一个标准的“集装箱”，彻底消灭了“这代码在我的机器上明明能跑啊”的玄学问题。
  • k8s (Kubernetes): 容器编排的绝对统治者（云原生操作系统）。如果 Docker 是集装箱，k8s 就是全自动的超级自动化港口，负责成千上万个容器的自动部署、弹性扩缩容和宕机自愈。
  • Protobuf (Protocol Buffers): Google 开发的二进制数据序列化协议。比传统的 JSON 和 XML 更小、更快、更省网络带宽，是现代微服务间高效 RPC 通信（如 gRPC）的首选数据格式。

Raft 算法因为其易于理解和极高的工程实现性，在现代分布式系统中几乎已经取代了晦涩难懂的 Paxos，成为了分布式共识算法的“工业标准”。

以下是目前主流的、正在使用 Raft 算法的知名项目，我将它们按应用领域进行了分类：

1. 分布式协调与元数据存储（核心中枢）

这是 Raft 最经典的应用场景，用于保证配置信息和集群状态的强一致性。

  • etcd: * 地位： Raft 算法最著名的工程实现。

  • 用途： Kubernetes (k8s) 的底层核心数据库，存储了整个集群的所有状态（如 Pod 位置、配置等）。可以说，没有 Raft，就没有今天稳定的 k8s。

  • Consul: * 用途： HashiCorp 公司开发的服务发现和配置管理工具。它使用 Raft 来管理数据中心的服务位置信息和健康状态。

2. 分布式数据库（存储基石）

现代“云原生”数据库（NewSQL）利用 Raft 来解决跨机器的数据复制和自动故障转移。

  • TiDB (TiKV): * 用途： 国内最著名的开源分布式关系型数据库。其底层存储引擎 TiKV 将数据切分成许多个 Region，每个 Region 都是一个独立的 Raft 组，从而实现了海量数据的强一致性分片。
  • CockroachDB: * 用途： 被称为“分布式数据库中的蟑螂”（极难杀死）。它在底层同样使用 Raft 协议来保证分布在不同城市的节点之间数据的一致性。
  • InfluxDB (Enterprise):
  • 用途： 领先的时序数据库，其集群版本利用 Raft 确保元数据和写操作的一致性。

3. 消息队列与流处理

为了保证消息“不丢失”且“顺序一致”，消息中间件开始转向 Raft。

  • Apache RocketMQ (Dledger): * 用途： 引入了 Dledger 模式，利用 Raft 来实现多副本间的 Leader 选举和数据同步，替代了传统的 Master-Slave 同步方案。
  • Redpanda:
  • 用途： 一个兼容 Kafka 协议的高性能流处理平台。它完全抛弃了 Zookeeper，底层原生集成了 Raft 协议，使得架构更加简洁。

4. 基础设施与服务治理

  • Docker Swarm: * 用途： Docker 自带的容器编排工具。它的经理节点（Manager Nodes）之间使用 Raft 来同步集群状态。
  • Baidu braft:
  • 用途： 百度开源的工业级 C++ Raft 实现，被广泛应用于百度内部的各种分布式系统中。

💡 为什么大家都选 Raft 而不是 Paxos？

  1. 强领导者模型： Raft 明确了 Leader 的权力，所有写请求必须经过 Leader，简化了逻辑。
  2. 状态机复制清晰： 将问题分解为 Leader 选举、日志复制和安全性三个子问题，工程师更容易写出无 Bug 的代码。
  3. 配置更改安全： Raft 提供了优雅的成员变更机制（比如从 3 台机器扩容到 5 台），而不会导致集群停摆。
