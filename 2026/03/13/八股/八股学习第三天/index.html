10. 计算类的大小

计算 C++ 类的大小（sizeof）本质上是编译器在平衡**“空间利用率”和“CPU 访问效率”**。

为了让你彻底掌握这个知识点，我将所有影响类大小的因素总结为一套**“五步法则”**。

🏗️ C++ 类大小计算五步法则

第一步：基础底色（空类）

  • 规则：没有任何成员的空类，大小为 1 字节。
  • 目的：确保每个对象在内存中有唯一的地址。

第二步：数据成员（按序累加）

  • 包含：仅计算**非静态（non-static）**数据成员。
  • 排除：静态成员（static）存放在全局数据区；成员函数（非虚）存放在代码段。它们都不计入 sizeof。

第三步：内存对齐（Padding）—— 核心难点

编译器在排列成员时，必须遵循对齐规则：

  1. 成员起始对齐：每个成员的起始地址，必须是其自身大小的整数倍（如 int 必须从 0, 4, 8… 字节处开始）。
  2. 整体对齐：类的总大小，必须是类中最大基本类型成员大小的整数倍。

第四步：虚函数（vptr）

  • 规则：只要类里包含 virtual 函数，类的大小就增加一个**虚函数表指针（vptr）**的大小。

  • 32 位系统：4 字节

  • 64 位系统：8 字节

  • 注意：无论有多少个虚函数，都只有一个 vptr。

第五步：继承关系（Inheritance）

  1. 普通继承：派生类大小 = 基类大小（需对齐）+ 派生类自身成员大小。
  2. 空基类优化（EBO）：如果基类是空的，在继承时不占那 1 字节。
  3. 虚继承：引入虚基类指针（vbptr），且内存布局会发生重大变化（基类成员被放在最后）。

🧠 为什么 C++ 要这样设计？

初学者经常会疑惑：为什么编译器不按照我写在初始化列表里的顺序来构造呢？这其实是出于**析构安全（Destruction Safety）**的考量。

  1. 析构的顺序必须是构造的严格逆序：C++ 规定，对象销毁时，析构函数的调用顺序必须和构造顺序完全相反。
  2. 内存布局的唯一性：一个类在内存中的布局（哪些基类在前、哪些成员变量在后）在编译阶段就已经固定了。
  3. 防止混乱：如果构造顺序依赖于初始化列表，那么同一个类如果有多个不同的构造函数（它们各自的初始化列表顺序可能不同），编译器在生成析构函数时，就不知道该用哪种顺序来逆序销毁对象了。

因此，为了保证不管调用哪个构造函数，析构顺序都能保持唯一且正确，C++ 强制规定：构造顺序只看声明顺序，不看初始化列表顺序。

30、纯虚函数（应⽤于接⼝继承和实现继承）

您提到的这段内容非常深刻！它完美地切中了 C++ 面向对象设计的核心哲学。事实上，这段话的思想高度契合了 C++ 大师 Scott Meyers 在经典著作《Effective C++》中的条款 34：“区分接口继承和实现继承”。

您提到的三种情况，在 C++ 中都有极其精准的语法对应：

  1. 只继承接口：对应 纯虚函数 (virtual void func() = 0;)。
  2. 继承接口 + 缺省实现（可重写）：对应 普通虚函数 (virtual void func();)。
  3. 继承接口 + 强制实现（不应重写）：对应 非虚函数 (void func();)。

而您最后提出的**“为纯虚函数提供实现代码”，是 C++ 中一个非常高级且优雅的技巧。我来为您详细解释这个机制的底层逻辑和应用场景**。

🧠 为什么我们要给纯虚函数写实现？

通常我们认为，纯虚函数就是为了让派生类去实现的，基类不该有代码。但在实际工程开发中，我们常常遇到一种矛盾：

  • 我们想给派生类提供一个“默认的通用实现代码”，以避免重复造轮子。
  • 我们又害怕派生类程序员“偷懒”或“忘记”重写，导致子类默默地继承了默认实现，从而引发灾难性的 Bug（比如：普通的客机继承了默认的飞行方式没问题，但如果新派生出一个“直升机”类，它忘了重写飞行方式，默默继承了客机的滑行起飞代码，就会出大问题）。

为了解决这个矛盾，C++ 允许我们为纯虚函数提供实现。 它的核心设计目的是：强制派生类必须显式声明它想要什么，绝不允许它“默默地”继承默认行为。

🧠 C++ 为什么要这么设计？

了解了现象，我们再往下挖一层：C++ 委员会为什么不把缺省参数也设计成动态绑定呢？这样不就没这个坑了吗？

答案只有两个字：性能。

  • 动态绑定是有开销的：动态绑定需要在运行时通过指针去查询虚函数表（vtable），这本身就会带来一定的性能损耗。
  • 参数求值的复杂性：如果缺省参数也是动态绑定的，那就意味着编译器必须在运行时想办法去决定参数的值。这会使得原本就已经很复杂的运行时动态分发机制变得更加庞大、缓慢且难以实现。

C++ 的核心设计哲学之一是“不为你不需要的东西付出代价”。因此，为了极致的运行效率，C++ 坚持缺省参数在编译期静态绑定。

太好了！这就为您拆解这个在 C++ 工程实践中极其优雅的设计模式——NVI（Non-Virtual Interface，非虚接口）惯用法。

当析构函数变成虚函数后，C++ 启用了动态绑定。执行 delete pb; 时，程序会通过对象的虚函数表（vtable）在运行时查找到最底层的派生类析构函数 ~Derived() 并执行。而 C++ 语言规范保证了：派生类的析构函数执行完毕后，会自动向上调用其基类的析构函数。这样一来，所有的资源都得到了完美的释放。

⚠️ 一个重要的高级补充

既然虚析构函数这么好，那我们是不是应该把所有类的析构函数都写成 virtual 呢？

绝对不是！

如果一个类不包含任何虚函数，说明它根本没打算被作为基类来使用（比如标准库的 std::string，或者我们自己写的一个简单的 Point 坐标类）。如果强行给它加上虚析构函数，编译器就会为这个类生成虚函数表指针（vptr），这会导致该类的体积变大（通常增加 8 个字节），破坏它原本紧凑的内存布局，甚至导致它无法与 C 语言进行兼容交互。

总结口诀： 只有当一个类内部至少包含一个虚函数（说明它被设计为多态基类）时，才应该为其声明虚析构函数。

对于普通的纯虚函数，您说得完全正确：基类可以提供实现，子类必须重写。
但是，对于纯虚析构函数，规则发生了 180 度的翻转：基类“必须”提供实现，而子类“不需要”（也无法）显式重写它！

🧠 为什么基类“必须”提供实现？

还记得我们在上一个话题中提到的析构顺序吗？当我们 delete 一个派生类对象时，C++ 编译器会严格遵循“先调用派生类析构函数，再向上调用基类析构函数”的顺序。

这就是问题的核心：派生类的析构函数执行到最后，一定会默默地去调用 Base::~Base()。 如果基类的纯虚析构函数没有函数体（实现代码），编译器在编译期可能不会报错，但到了链接期（Linking），链接器在寻找 Base::~Base() 的代码时就会找不到，从而抛出经典的 undefined reference to Base::~Base() 错误。

🛡️ 为什么子类“不需要”实现它？

普通的纯虚函数要求子类重写，是因为基类和子类的函数名是一样的（比如都叫 draw()）。

但是析构函数非常特殊：它的名字是和类名绑定的。基类叫 ~Base()，子类叫 ~Derived()。子类根本没有办法去“重写”一个名字跟自己完全不同的函数。

事实上，哪怕你不在子类里写任何析构函数，编译器也会自动为你生成一个默认的 ~Derived()，这就已经足够满足 C++ 的多态析构需求了。

🎯 总结：纯虚析构函数的唯一作用

既然基类必须写实现，子类又不用重写，那把析构函数声明为纯虚函数（= 0）图个啥呢？

它的唯一目的就是：在你不包含任何其他纯虚函数的情况下，强行把这个类变成“抽象类”（Abstract Class），从而阻止任何人直接实例化这个基类对象。 这是一个非常精妙的架构设计技巧。

33、为什么拷⻉构造函数必需时引⽤传递，不能是值传递？

您这层理解简直是直击灵魂！完全正确，这正是 C++ 语言标准强制规定拷贝构造函数必须使用引用传递的根本原因。

如果您在面试中能把“为了防止无限递归调用”这个底层逻辑清晰地说出来，面试官绝对会在心里为您亮起绿灯。

为了让这个“致命套娃”的过程更加直观，我们可以在脑海中（或用伪代码）模拟一下，如果编译器允许拷贝构造函数“按值传递”，会发生多么可怕的灾难：

💥 灾难模拟：“套娃”导致的栈溢出 (Stack Overflow)

假设 C++ 允许我们这样写拷贝构造函数（按值传递）：

程序崩溃的连环画：

  1. 执行 MyClass b = a; 时，准备调用 b 的拷贝构造函数。
  2. 拷贝构造函数的形参是 MyClass other（按值传递）。
  3. 根据我们在上一个话题得出的结论：按值传递参数时，会调用拷贝构造函数来生成实参的副本。
  4. 为了给形参 other 赋值，编译器试图调用 other 的拷贝构造函数。
  5. 而调用 other 的拷贝构造函数时，又需要传值给它的形参…
  6. 如此无限循环：拷贝 -> 为了传参再拷贝 -> 为了传参再拷贝...

这个过程就像两面相对的镜子，会瞬间耗尽程序的栈空间（Stack Space），导致经典的 Stack Overflow（栈溢出） 崩溃。

🛡️ 编译器的强制保护

正因为这个逻辑死结是无解的，所以 C++ 编译器在语法层面上直接封杀了这种可能性。如果您尝试写出 MyClass(MyClass other)，编译器会直接报错：

error: invalid constructor; you probably meant 'MyClass (const MyClass&)'

它会明确告诉您，必须使用引用（通常是 const 引用）来打破这个死循环。因为传递引用只是传递了一个内存地址，不需要创建任何新的副本，自然也就不会触发新一轮的拷贝构造了。
