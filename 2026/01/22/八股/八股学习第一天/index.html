一、说一下你理解的C++中的四种智能指针

1、auto_ptr（C++98的方案，C11已抛弃）采用所有权模式。
auto_ptr 的赋值 / 拷贝会粗暴转移所有权，原对象指针被置空，编译不报错但运行时访问原对象会崩溃；
核心原因是其赋值运算符会把原对象的内部指针设为 nullptr，导致后续解引用空指针；
C++11 用 unique_ptr 替代 auto_ptr，通过禁用拷贝、支持移动语义解决了这个致命缺陷，是更安全的选择。
2、unique_ptr（替换auto_ptr）采用独占模式，即某个时刻只能有一个智能指针指向该对象。
3、shared_ptr（共享型，强引用）
shared_ptr实现共享式拥有概念，多个智能指针可以指向相同对象，该对象和其相关资源会在“最后一个引用被销毁”时候释放。从名字share就可以看出了资源可以被多个指针共享，它使用引用计数机制来表明资源被多个指针共享。
可以通过成员函数use_count()来查看资源的所有者个数，除了可以通过new来构造，还可以通过传递auto_ptr, unique_ptr,weak_ptr来构造。当我们调用release()时，当前指针会释放资源所有权，计数减少。当计数等于0时，资源会被释放。
shared_ptr是为了解决auto_ptr在对象所有权上的局限性(auto_ptr是独占的)，在使用引用计数的机制上提供了可以共享所有权的智能指针。
4、weak_ptr（弱引用）
weak_ptr是一种不控制对象生命周期的智能指针，它指向一个shared_ptr管理的对象。进行该对象的内存管理的是那个强引用的shared_ptr。
weak_ptr只是提供了对管理对象的一个访问段。weak_ptr设计的目的是为配合shared_ptr引用的一种智能指针来协助shared_ptr操作，它只可以从一个shared_ptr或另一个weak_ptr对象构造，它的构造和析构不会引起引用计数的增加或减少。
weak_ptr是用来解决shared_ptr相互引用时的死锁问题，如果说两个shared_ptr相互引用，那么这两个指针的引用计数永远不可能下降为0，也就是资源永远不会释放。它是对对象的一种弱引用，不会增加对象的引用计数，和shared_ptr之间可以相互转化，shared_ptr可以直接赋值给它，它可以通过调用lock函数来获得shared_ptr。

2、C++中内存分配情况

1、C++程序运行时的核心内存分区（5大区域）

C++程序启动后，操作系统会为其分配一段内存空间，按用途分为5个核心区域，各自有明确的生命周期和使用规则：

分区名称 存储内容 生命周期 分配/释放方式 典型示例
栈（Stack） 局部变量、函数参数、返回值 随作用域（{}）创建/销毁 编译器自动分配/释放（无需手动） int a = 10;、函数参数void f(int b)
堆（Heap） 动态分配的对象/数据 手动控制（直到程序结束） new/delete、malloc/free手动管理 int* p = new int(20);、char* buf = (char*)malloc(100);
全局/静态区 全局变量、static静态变量 程序启动创建，退出销毁 编译器自动分配/释放 int g_num = 0;、static int s_num = 1;
常量区（只读） 字符串常量、const常量（全局） 程序启动创建，退出销毁 编译器自动分配/释放（只读） const char* str = "hello";、const int c_num = 100;
代码区 程序的二进制指令（函数、逻辑） 程序启动加载，退出释放 操作系统加载时分配 所有函数（如main()、f()）的执行代码

    • 分配速度极快（编译器直接操作栈指针），无需手动管理，效率最高。

3、容易混淆的点（新手避坑）

  1. 栈 vs 堆：生命周期

    • 栈变量：void f() { int a = 1; } → 函数结束，a 立即销毁；
    • 堆变量：void f() { int* p = new int(1); } → 函数结束，p 销毁，但堆内存仍存在（泄漏）。

  2. static变量的存储
    static int a = 1; 不管定义在函数内还是全局，都存在全局/静态区，而非栈！生命周期是整个程序运行期。

  3. 智能指针的内存归属
    unique_ptr<int> p(new int(1)); → 智能指针对象p（栈），管理的内存（堆），p 析构时自动释放堆内存。

总结

  1. C++内存核心分为栈、堆、全局/静态区、常量区、代码区，栈/堆是日常开发最常接触的；
  2. 栈内存自动管理（效率高、大小有限），堆内存需手动new/delete（灵活、易泄漏）；
  3. C++风格的new/delete会调用构造/析构函数，是面向对象编程的首选，malloc/free仅适用于兼容C代码场景。

3、C++中的指针参数传递和引用参数传递

如果面试官问：

引用一定是指针实现的吗？

标准回答：

C++标准只规定引用是变量的别名，没有规定具体实现方式。
在大多数编译器中，引用通常通过指针实现，在函数调用时会传递变量地址。但在简单场景下编译器可能直接将引用优化为变量别名，因此引用不一定占用额外内存。

指针参数传递本质上是值传递，它所传递的是一个地址值。值传递过程中，被调函数的形式参数作为被调函数的局部变量处理，会在栈中开辟内存空间以存放由主调函数传递进来的实参值，从而形成了实参的一个副本（替身）。值传递的特点是，被调函数对形式参数的任何操作都是作为局部变量进行的，不会影响主调函数的实参变量的值（形参指针变了，实参指针不会变）。
引用参数传递过程中，被调函数的形式参数也作为局部变量在栈中开辟了内存空间，但是这时存放的是由主调函数放进来的实参变量的地址。被调函数对形参（本体）的任何操作都被处理成间接寻址，即通过栈中存放的地址访问主调函数中的实参变量（根据别名找到主调函数中的本体）。因此，被调函数对形参的任何操作都会影响主调函数中的实参变量。

C++中const和static关键字

static作用：控制变量的存储方式和可见性。

作用一：修饰局部变量：

用static关键字修饰的话，该变量便会存放在静态数据区，其生命周期会一直延续到整个程序执行结束。但是要注意的是，虽然用static对局部变量进行修饰之后，其生命周期以及存储空间发生了变化，但其作用域并没有改变，作用域还是限制在其语句块。

作用二：修饰全局变量：

对于一个全局变量，它既可以在本文件中被访问到，也可以在同一个程序中其它源文件被访问(添加extern关键字声明即可)。用static对全局变量进行修饰改变了其作用域范围，由原来的整个程序可访问变成了本文件可访问。

作用三：修饰函数：

用static修饰函数，情况和修饰全局变量类似，也是改变了函数的作用域。在 C++ 中，普通全局函数默认具有 external linkage，可以被其他源文件访问。
使用 static 修饰后，函数的链接属性变为 internal linkage，使其只在当前源文件可见，从而避免函数名冲突并实现模块内部封装

作用四：修饰类：

如果C++中对类中的某个函数用static修饰，则表示该函数属于一个类而不是属于此类的任何特定对象；如果对类中的某个变量进行static修饰，则表示该变量以及所有的对象所有，存储空间中只存在一个副本，可以通过；类和对象去调用。

作用五：类成员/类函数声明static

函数体内static变量的作用范围为该函数体，不同于auto变量，该变量的内存只被分配一次，因此其值在下次调用时仍维持上次的值；
在模块内的static全局变量可以被模块内所有函数访问，但不能被模块外其它函数访问；
在模块内的static函数只可被这个模块内的其它函数调用，这个函数的使用范围被限制在声明它的模块内；
在类中的static成员变量属于整个类所拥有，对类的所有对象只有一份拷贝；
在类中的static成员函数属于整个类所拥有，这个函数不接收this指针，因为只能访问类的static成员变量。static类对象必须要在类外进行初始化，static修饰的变量先于对象存在，所以static修饰的变量要在类外初始化；
由于static修饰的类成员属于类，不属于对象，因此static类成员函数是没有this指针，this指针是指向本对象的指针，正因为没有this指针，所以static类成员函数不能访问非static的类成员，只能访问static修饰的类成员；
static成员函数不能被virtual修饰，static成员不属于任何对象或实例，所以加上virtual没有任何实际意义；静态成员函数没有this指针，虚函数的实现是为每一个对象分配一个vptr指针，vptr是通过this指针调用的，所以不能为virtual；虚函数的调用关系，this->vptr->ctable->virtual function。

const作用：

const修饰变量，表示常量，不可修改；
const修饰指针，分为两种情况：
1. const修饰指针指向的内容，表示指针指向的内容不可修改；
2. const修饰指针本身，表示指针不可修改；

const修饰成员函数，表示成员函数内不能修改成员变量（除了mutable修饰的成员变量），可以访问所有成员变量（包括mutable修饰的成员变量）；
const修饰对象，表示对象内的所有成员变量都是const，不可修改，只能访问；

1. const 成员变量

  • 核心特性： 在某个对象的生命周期内是常量，但对整个类而言，不同对象可以有不同的值。
  • 初始化方式： 必须在构造函数的初始化列表中进行初始化。

  • ⚠️ 现代 C++ 补充： 从 C++11 开始，是可以在类声明中直接为 const 数据成员提供默认初始值的（就地初始化）。但如果每个对象在创建时需要赋予不同的常量值，依然必须依赖构造函数的初始化列表。

2. const 成员函数

  • 核心目的： 承诺不修改调用该函数的对象的状态（即不修改成员变量）。
  • 与 static 互斥： const 修饰的是隐藏的 this 指针，而 static 静态成员函数属于类本身，根本没有 this 指针，因此两者不能同时使用。
  • 底层原理（隐式 this 指针）：
  • 对于普通成员函数，隐藏的 this 指针类型是：ClassName* const this（指针本身的指向不能变，但指向的对象内容可以变）。
  • 对于 const 成员函数，隐藏的 this 指针类型变成了：const ClassName* const this（不仅指针指向不能变，通过指针修改对象内容也不允许）。

3. const 对象（常量对象）及其调用规则

  • 常量对象： 只能调用 const 成员函数，绝对不能调用非 const 成员函数。因为非 const 函数有修改对象状态的风险，编译器必须将这种风险扼杀在摇篮里。
  • 非常量对象： 既可以调用普通的非 const 成员函数，也可以调用 const 成员函数。

4. 突破限制与全局常量

  • mutable 关键字： 如果在 const 成员函数中，迫切需要修改某个用于内部状态追踪的变量（例如：互斥锁、缓存标记、调用次数统计），可以用 mutable 修饰该成员变量。
  • 类级别的绝对常量： 如果想在整个类中建立所有对象共享的恒定常量，应使用枚举常量（enum）或 static const。

  • 💡 现代 C++ 补充： 在现代 C++ 中，定义类级别的编译期常量，更推荐使用 static constexpr。

很多 C++ 学习者只记住了“常量对象只能调用常量函数”的表面现象，而您已经透彻理解了编译器在背后是如何通过指针的 const 属性来强制保证这一点的。

为了让这个知识点更加立体，我结合您的笔记，从权限控制（C++ 隐式类型转换机制）的角度为您做一个清晰的梳理：

1. 隐式 this 指针的本质

如您所述，任何非静态成员函数在被调用时，编译器都会偷偷塞进去一个 this 指针参数：

  • 普通成员函数： 隐式参数类型是 test* const this。这代表 this 指针的指向不能变（你不能让它指到别的对象上去），但指向的内容可以变。
  • const 成员函数： 隐式参数类型变成了 const test* const this。这代表不仅指向不能变，指向的内容也绝对不能变。

2. 为什么“常量对象不能调用非 const 函数”？（权限放大的错误）

假设有一个常量对象 const test obj;。

  • obj 的内存地址类型本身就带有 const 属性，即 const test*。
  • 如果 obj 去调用普通成员函数，编译器需要把一个 const test* 的地址，传递给类型为 test* const this 的形参。
  • 核心冲突： 这意味着试图把一个只读的地址赋值给一个可读可写的指针。在 C++ 中，隐式丢弃 const 限定符（权限放大）是严格禁止的，因此编译器会直接报错，从而保护了常量对象不被意外修改。

3. 为什么“非常量对象可以调用 const 函数”？（权限缩小的安全）

假设有一个非常量对象 test obj;。

  • obj 的内存地址类型是 test*（可读可写）。
  • 如果 obj 去调用 const 成员函数，编译器需要把 test* 传递给类型为 const test* const this 的形参。
  • 核心原理： 编译器允许将一个非 const 指针隐式转换为 const 指针。这相当于你把一个可读可写的权限主动缩小为只读权限。这种操作绝对安全，所以非常量对象既能调用普通函数，也能调用常量函数。
